# Building a company operating system

A typical engagement takes four to eight weeks for the first production workflows, depending on tool access, risk, and review speed.

## 1. Discovery and workflow selection · 3–5 days

Map where work enters, who owns decisions, what routinely stalls, which sources are authoritative, and what success can be measured. Select one or two valuable workflows with bounded effects rather than attempting to automate the whole company at once.

## 2. Tool, data, and risk audit · 3–7 days

Inventory systems, APIs, identity, roles, data classifications, hosting constraints, vendor agreements, retention, incident routing, and existing automation. Define which reads and writes are allowed before building prompts.

## 3. Pack selection or build · 4–10 days

Start from an existing industry pattern or build a new pack. The new pack captures terminology, stricter governance, skills, integration recipes, and safe memory structure. The reusable engine remains unchanged.

## 4. Company tailoring and implementation · 1–3 weeks

Instantiate the agent contract, roles, voice, policies, tool operation allowlists, reviewed memories, workflow versions, approval owners, mock fixtures, and audit views. Build against sandbox or mock tools first; connect production only after deterministic validation.

## 5. Pilot, handoff, and support · 1–2 weeks

Run dry runs, failure drills, permission tests, human approval tests, and measured pilot traffic. Handoff includes the customer-owned repository, deployment, credentials inventory, operating documentation, training, and rollback procedure. Ongoing support can add workflows, tools, providers, and memory review without making the client dependent on a closed platform.
